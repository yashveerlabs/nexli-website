## EdTech Interoperability Standards: A Guide for Indian School Buyers

Interoperability standards are the technical specifications that allow different educational technology products to work together. When two products from different vendors implement the same standard, they can exchange data and coordinate without custom development work. When products do not implement common standards, every connection between them requires bespoke engineering.

Understanding the key standards helps Indian school leaders ask better questions during EdTech procurement and avoid buying products that create integration problems and vendor dependency.

### Why Interoperability Matters for Indian Schools

Indian schools are increasingly assembling technology stacks from multiple vendors: one vendor for school management, another for digital learning content, another for assessment, another for communication. Without interoperability standards, each connection between these systems requires custom integration work, and the school becomes dependent on vendors maintaining those custom connections.

With interoperability standards, the school can swap out one component (replace the assessment tool) without rebuilding connections to every other system, because all systems speak the same standard.

This is also a procurement signal. A vendor that claims to support industry standards but cannot demonstrate compliance is a risk. A vendor that is actively certified by the relevant standards body has invested in making their product interoperable.

### The Key Standards and What They Do

**IMS Global Learning Consortium**

IMS Global (now operating as 1EdTech) is the standards organization that maintains several of the most important EdTech interoperability standards. IMS Global also runs a certification program that tests and verifies whether products correctly implement the specifications they claim to support.

**LTI 1.3 (Learning Tools Interoperability 1.3)**

LTI allows a tool (an external learning app, an assessment platform, a simulation) to be embedded within an LMS or portal and exchange information about the user and their results.

Specifically, LTI enables:

Single sign-on context passing: The student or teacher who is already logged into the school portal launches the external tool without needing a separate login. The portal passes the user's identity to the tool automatically.

Grade passback: When the student completes an activity in the external tool, the score is passed back to the portal's gradebook automatically.

Roster provisioning: The external tool can receive the class list and enrollment information from the portal.

LTI 1.3 (the current version) is significantly more secure than earlier versions because it uses public key cryptography rather than shared secrets. Schools procuring new tools should specify LTI 1.3 specifically.

**OneRoster**

OneRoster is a standard for exchanging student enrollment and roster data between systems. A school's ERP holds the authoritative class list: which students are in which sections, which teachers teach which classes. An LMS or learning tool needs this information to set up the right classes and enroll the right students.

Without OneRoster, schools manually upload CSV files to each tool at the start of each term, and manual updates when students change sections or teachers are assigned differently. With OneRoster, the LMS or learning tool receives automatic updates from the ERP.

OneRoster 1.2 is the current specification. It defines the data schema for schools, courses, classes, users, enrollments, and grades in a standard way that any compliant system can consume.

**xAPI (Experience API)**

xAPI, also called Tin Can API, is a standard for recording learning activities in a structured way. An xAPI statement looks like: "Student A completed Module 3 of Biology, scoring 85%, on 2026-06-15 at 10:23am."

These statements can be generated by any xAPI-compatible learning tool and sent to a Learning Record Store (LRS), which aggregates them across all tools. This creates a consolidated learning record that spans multiple applications.

xAPI is particularly useful for schools using multiple digital learning tools and wanting a unified picture of student learning activity across all of them, without manually extracting data from each tool separately.

**SCORM (Sharable Content Object Reference Model)**

SCORM is an older standard (most widely used version is SCORM 1.2, from 2001) for packaging e-learning content so that it works in any compatible LMS. A SCORM package created by an instructional designer in one tool can be imported into any SCORM-compatible LMS.

SCORM is still widely used because there is a large installed base of SCORM content. If a school has existing SCORM content libraries or is buying content from publishers who provide SCORM packages, the LMS must support SCORM.

xAPI is more capable than SCORM (it can track a wider range of learning activities and works outside of a browser LMS environment) but has not yet replaced SCORM in the installed base.

**QTI (Question and Test Interoperability)**

QTI is a standard for representing assessment questions and results. A question bank created in one QTI-compliant tool can be exported and imported into another QTI-compliant tool.

For schools that invest in building question banks (which takes significant teacher time), QTI portability means that investment is not lost if the assessment platform changes.

### Asking Vendors About Standards Compliance

**Questions to ask:**

"Do you support LTI 1.3?" (If you are evaluating an LMS or a learning tool to connect to an LMS)

"Are you certified by IMS Global or 1EdTech for the standards you claim to support?"

"Do you support OneRoster for automated roster sync?"

"What xAPI or SCORM versions does your platform support?"

"Can we see a live demonstration of an LTI launch from another system?"

A vendor who answers these questions confidently and can demonstrate compliance is more trustworthy than one who says "yes" to everything but cannot demonstrate it or explain what the standard does.

**Red flags:**

"We have our own integration approach that works better than standards" often means the vendor has built a proprietary connection to specific partners and cannot integrate with anything outside that partner ecosystem.

"We support that standard" followed by inability to name the specific version or demonstrate it in practice.

"Standards don't matter for Indian schools" because standards adoption is lower in India does not mean it does not matter. It means Indian schools that understand standards can make better procurement decisions than those that do not.

## Frequently Asked Questions

**Q: Are IMS Global standards widely adopted in India?**
A: Adoption is growing but uneven. International EdTech vendors (global LMS platforms, large content publishers) generally support IMS Global standards. Many Indian EdTech startups have not yet implemented these standards. Indian schools that understand standards are in a position to ask better questions and favor vendors who have invested in interoperability.

**Q: Is IMS Global certification expensive for vendors?**
A: IMS Global certification involves a testing and verification process that takes vendor time. It is not free, which is why some smaller vendors have not pursued it. However, IMS Global certification is a meaningful quality signal because it involves independent testing rather than self-assessment.

**Q: Can a school test whether an LTI integration actually works before buying?**
A: Yes. Ask the vendor to demonstrate a live LTI launch from your existing portal into their tool in a sandbox environment. This is a reasonable pre-purchase request and any vendor with genuine LTI support should be able to accommodate it.

**Q: Does xAPI replace the need for a gradebook in the LMS?**
A: No. xAPI records learning activity statements in a Learning Record Store. The LRS is separate from a gradebook. A gradebook presents structured grade data for courses and assessments. Both serve different purposes. Many schools use xAPI for detailed learning analytics while still maintaining a gradebook for formal grade reporting.

**Q: Should schools require government-aligned standards for UDISE+ and other regulatory submissions?**
A: Yes. Government data submissions (UDISE+, state education board data) use specific data formats defined by those authorities. Schools should verify that their ERP can generate reports in the exact formats required for regulatory submissions, not just "we can export the data" in a generic format. Verify with reference to the official government data submission templates.
