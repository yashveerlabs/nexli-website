## ERP Customization Options: Configuration vs. Code for Schools

**"The system can be customized to your needs" is one of the most common sales statements, and one of the most meaningless without context. Customization through configuration (no code, immediate, reversible) is fundamentally different from customization through development (expensive, time-consuming, creates technical debt). Understanding this difference prevents months of frustration.**

---

## Configuration: The Right Kind of Customization

Configuration means changing system behavior through the user interface or admin settings, without writing code.

**Examples of configuration in a school ERP:**
- Adding a new role (DPO, Academic Coordinator Tier 2) and setting permissions for each module
- Defining a new fee structure (₹60,000 per year, quarterly installments, 10% sibling discount)
- Changing the grading scale (switching from CBSE 9-point to a custom A/B/C/D scale)
- Creating a new announcement type or category
- Setting attendance thresholds (alert parents at 80%, flag at 75%)
- Adding custom fields to student profiles (e.g., "Transport Zone")

Configuration changes:
- Done by the school admin, not the vendor
- Take minutes to hours, not weeks
- Are reversible, you can undo a configuration change
- Do not affect the system's ability to receive updates

**The ideal school ERP does 90% of customization through configuration.**

---

## Development: When Code Is Required

Some requirements genuinely need development, new features, complex integrations, or functionality the system was not designed to support.

**Examples that require development:**
- A completely new module (e.g., an alumni fundraising platform)
- Integration with a custom in-house system
- A regulatory report in a non-standard format specific to one state board
- Complex conditional logic that the UI does not support

Development creates technical debt. When the ERP vendor releases an update, custom code may conflict. You pay for the initial development, then pay again every time it needs to be updated to stay compatible.

**Red flag:** If a vendor says "We can customize that for you" when you ask for a basic feature like configurable roles or a custom fee structure, the system is not designed for flexibility. You will keep paying for development every time your needs evolve.

---

## What to Look for in Customizable ERPs

### Configurable Roles

The role system should be fully configurable without code. You should be able to:
- Create a new role (e.g., "Campus Coordinator")
- Set what modules this role can access (e.g., attendance, reports, not fees)
- Set what actions they can take in each module (View only, or View + Edit)
- Deploy the new role immediately

Nexli supports 118+ roles through configuration. Adding a new role requires no vendor involvement.

### Configurable Fee Structures

Fee structures change every academic year. You should be able to:
- Add new fee heads (e.g., "Activity Fee 2026")
- Change amounts per class
- Modify installment schedules
- Apply new concession types

All through the admin interface, not through a support ticket.

### Configurable Grading and Report Cards

Different boards have different grading scales. Different classes within a school may use different scales. The system should let you configure this per class group.

### Custom Fields

Occasionally, a school needs to track something the ERP does not have a built-in field for. A good system allows custom fields on student profiles, staff profiles, or other records.

## Development Requests: Ask These Questions

Before agreeing to any custom development:

1. "How long will this take?" If more than 2 weeks, is it worth the delay?
2. "What does it cost?" Get this in writing, not as an estimate.
3. "Will this be maintained in future updates, or will I need to pay for updates?"
4. "Why isn't this handled by configuration?" Understand why development is needed.

## FAQ

**Q: What if we need a feature that does not exist anywhere in the system?**
A: Evaluate whether the gap is a true requirement or a preference. If it is critical, ask about development cost and timeline. Also ask about the roadmap, is this on the vendor's plan for the next 6 months?

**Q: Can we configure the system ourselves or do we need vendor help?**
A: Configuration should be self-service. If every configuration change requires a support ticket and vendor action, the system is not actually configurable, it just means the vendor does custom development and calls it configuration.

**Q: What happens to customizations when the ERP is updated?**
A: Configuration-based customizations survive updates. Development-based customizations may need to be reviewed and updated with each major ERP release. Factor this maintenance cost into development decisions.

**Q: Is there a limit to how many custom fields we can add?**
A: Reasonable limits exist in most systems (e.g., 10 custom fields per record type). Ask vendors about limits and whether they apply to your use case.

**Q: How do we request a new feature vs. custom development?**
A: New features in the product roadmap are available to all customers at no extra charge. Custom development for your school's specific need is chargeable. The vendor should distinguish these clearly.
