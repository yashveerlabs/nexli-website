## Attendance Module Comparison: What Your School ERP Must Do

**Attendance is the most frequently used module in any school ERP. Teachers mark it every day, parents check it every week, and the principal reviews it every month. A weak attendance module creates more work than it saves, slow loading, confusing UI, or missing features force staff to maintain parallel paper registers. This guide explains what the attendance module of a school ERP must do.**

---

## Two Types of Attendance: Both Are Required

Most schools need two distinct attendance systems that must work together:

**Period-wise (subject-wise) attendance:**
- Marked by the subject teacher at the start of each period
- Tracks which students attended each subject, not just the day
- Required for CBSE and most other boards (minimum 75% attendance per subject)
- Generates subject-wise attendance reports

**Daily consolidated attendance:**
- Marked by the class teacher, typically during the first period or morning assembly
- Single Present/Absent/Late/Leave status for the school day
- Feeds into the overall attendance percentage used for UDISE+ and compliance reporting
- The official record for end-of-day parent notifications

Systems that offer only one type create gaps. Schools that need both must either use two systems or maintain a paper register for the missing type.

## Core Features Every Attendance Module Must Have

### 1. Configurable Attendance Status Types

Minimum required statuses:
- Present
- Absent
- Late (with arrival time note)
- Leave (pre-approved, type: medical/personal/family/event)
- Duty (student on school duty: sports day, inter-school competition)

Systems that only support Present/Absent force teachers to use notes for nuances, which creates manual reconciliation work for the admin team.

### 2. 75% Threshold Tracking and Alerts

CBSE mandates 75% minimum attendance per subject for eligibility for board exams. An ERP should:
- Calculate running attendance percentage per student per subject in real time
- Flag students who cross 80% threshold (early warning) and 75% threshold (at-risk)
- Send automated alerts to parents when a student crosses the at-risk threshold
- Generate a list of students at risk per class, available to the Principal and HOD

Without automated threshold tracking, the HOD manually calculates percentages at the end of each term, a slow, error-prone process.

### 3. Mobile-Optimized Marking Interface

Teachers should be able to mark attendance on a phone in under 60 seconds for a class of 40 students. The interface should:
- Load the student list for the teacher's assigned class automatically (no searching)
- Allow bulk-marking (all Present, then mark exceptions)
- Support offline marking with sync when internet returns
- Confirm submission with a visual indicator

### 4. Biometric and RFID Integration

Optional but important for schools with biometric devices or RFID cards:
- Biometric device (fingerprint or face recognition) records attendance automatically
- Teacher can see who has biometrically checked in before the period starts
- Manual override available for devices that fail or for students without biometric registration
- RFID cards can mark bus boarding attendance separately from classroom attendance

### 5. Correction and Override Workflow

Attendance marked incorrectly needs a correction mechanism with audit trail:
- Teacher submits correction request (student marked absent but was present)
- HOD or admin reviews and approves/rejects
- Original entry and correction are both logged (audit trail for parent disputes)

Systems with direct edit (no audit trail) create disputes when parents contest attendance records.

### 6. Attendance Reports

Standard reports that must be available:
- Daily attendance register per class (present/absent count, names)
- Monthly summary per student (attendance percentage, subject-wise)
- Defaulter list (students below 75% threshold) per class
- Comparative report (last month vs. this month)
- Export to Excel/PDF for board submissions

### 7. UDISE+ Data Feed

Annual school data submissions to UDISE+ require attendance statistics. The ERP should pre-populate UDISE+ attendance fields from the attendance module data, not require manual re-entry.

## Optional But Valuable Features

**Bus attendance:** RFID or manual marking of students boarding the school bus, feeds transport module and generates parent alerts ("Your child has boarded the bus").

**Hostel roll-call integration:** Morning and night roll-call in the hostel linked to the main attendance system. Hostel absentees trigger alerts to the warden.

**Gamification:** Attendance streaks (7 days perfect attendance, 30 days perfect attendance) feed into student achievement badges. Schools that implement this report improved regular attendance among students who were previously borderline.

## FAQ

**Q: Can teachers mark attendance from their personal phones?**
A: Yes, if the system is mobile-first and web-based. Teachers open the ERP on their phone browser, log in, and see their assigned class attendance sheet. No separate device needed.

**Q: What if two teachers teach the same class (e.g., language and maths in one period)?**
A: Period-wise attendance is linked to the subject and teacher, not just the class. Each teacher marks their own subject attendance. The system prevents double-marking for the same period.

**Q: How do parent notifications work for attendance?**
A: When a student is marked absent, an automated notification goes to the parent (app, SMS, or WhatsApp depending on the system). Some systems allow real-time notification; others batch-notify at end of day. Ask vendors which model they use.

**Q: Can the principal see attendance across all classes in one view?**
A: Yes, in any well-designed system. The Principal dashboard shows today's attendance rate school-wide, with drill-down to individual classes and flagged absentees.

**Q: What happens to attendance data during internet outages?**
A: Mobile-first systems cache the class roster locally. Teachers mark attendance offline. When the device reconnects, records sync to the server. Data is not lost.
