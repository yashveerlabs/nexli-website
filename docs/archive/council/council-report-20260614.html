NEXLI — LLM Council Verdict

Implementation vs Master Specification · 5 advisors → anonymized peer review → chairman · 2026-06-14 · build green @603

Question: Audit NEXLI's current implementation against its Master Specification, operational realism, role/workflow ownership, permission architecture, and product vision. Where is it missing / incorrect / partial / deviating, and what corrections are justified? Goal: architectural correctness + operational realism + vision alignment before testing — not feature expansion.

Chairman's Verdict

Where the council agrees

The #1 issue is the security boundary. UI-level permission scoping is not real enforcement; on Spark (no Cloud Functions) the only server-side boundary is Firestore rules. Three advisors converged independently and peer review ranked them strongest (4/5). This is a present blocker, not a "P9/future" item — testing a UI-gated app validates nothing about who can read the database — and it is fixable on Spark for free (rules need no Blaze).

Where the council clashes

What to do first. The Executor says "lock the rules (1–3 days), defer everything else." The Outsider says the real trust-breaker is that parents/students have no way to message a teacher (the spec's #1 promise). The Expansionist says the biggest under-build is the stubbed Intelligence layer (AI panels are hardcoded mock data, ~60% of which needs no LLM). Peer review sided with the Executor and flagged the other two as "optimizing cosmetics over an open data layer." They aren't really competing — they answer different questions (most dangerous vs most adoption-breaking vs most vision-breaking) and are sequential.

Blind spots the peer review caught

  • Can the data model even be scoped by rules? Verified: docs carry schoolId + studentId/sectionId/staffId → the fix is additive per-collection rules, not a data migration.
  • RBAC-in-rules needs no Blaze (rules already get() the member doc); but nobody verified the rules are actually deployed.
  • The spec itself mandates Spark-impossible features (Cloud-Function account suspension, enforced MFA, SMS fan-out) — it should be formally reconciled to the Spark tier + server-enforced trust model, or every future build re-introduces the gap.
Chairman correction (post-verification of firestore.rules): the advisors who said "no rules / every collection is a wildcard / all UI theatre" overstated it. Rules do exist: deny-by-default, structurally tenant-isolated (no cross-tenant), with strict role allowlists on the most sensitive collections (medical, immunization, counseling, POCSO, grievances) + members/settings/append-only audit. But the substance holds: the coarse default rule (isActiveMember reads / isStaff writes) governs everything not on the restricted list — and that list omits fees, payroll, salary_structures, assessment_results, students (Aadhaar/guardians), staff (PAN), iep_plans, therapy_logs, consent_records, attendance. So a parent or student can read the whole school's marks, fees, payroll, salaries and Aadhaar, and any staff (bus driver/canteen) can write them. That is the real, DPDP-relevant gap.

The recommendation

Treat the Firestore rules as the next unit of work — but precisely. Tighten the default rule with explicit per-collection matches for the sensitive collections (role allowlists like the existing medical rule + own-record scoping via studentId for parents/students), verify the rules are deployed, and emulator-test against teacher-cross-section / parent / accountant / driver before any human resumes role-by-role testing. Then, as fast-follows (not blockers): give parents/students a real Messages entry point wired to the existing recipient policy, relabel "Communication"→"Notices" vs "Messages", and split the analytics panels into computed-now vs provider-later. Finally, reconcile the spec so it stops asserting protections the Spark tier can't run.

The one thing to do first

Verify the deployed rules, then add explicit restricted-collection rules for fees · payroll · salary_structures · assessment_results · students · staff · iep_plans · therapy_logs · consent_records · attendance (role allowlists + parent/student own-record scoping), and emulator-test before any further role testing.

Full advisor responses

The Contrarian — what fails

The central security claim is false. Spec §10 promises "no bug can expose cross-school data" + per-role allowlists at the rules layer; the 50-role engine and scoped grants are UI theatre. Concrete leaks the UI can't patch: assessment_results ship every classmate's marks; event_registrations expose all names — a live DPDP/§7.4 violation. Second gap: §7 communication spec unbuilt (matrix, banned patterns = POCSO, SLA, escalation; nav has communication/messages with no permission gate). Third: spec bakes in Spark-impossible capabilities (CF account suspension, MFA, SMS). Priority: rules-as-boundary before testing; then §7 engine; then reconcile spec to Spark.

The First Principles Thinker — what are we solving

The spec's mission (§1.2/§1.6) is to replace WhatsApp-and-registers with something trustworthy — trust is an enforced boundary ("least privilege is the foundation… non-negotiable"). The foundation rests on the wrong assumption: that the client can be trusted to scope itself. On Spark the only real boundary is Firestore rules; today the trust model is a .includes() on the attacker's machine — the inverse of the vision. The team is optimizing the wrong thing: 40 modules/50 roles above a tier where correctness can't be enforced. The next unit of work is firestore.rules, not module #41.

The Expansionist — the upside being missed

The spec promises five layers (§11.2); four shipped. The Intelligence layer is a painted backdrop — every AI surface is hardcoded mock data. Yet half the §8 AI catalogue needs no LLM: at-risk flags (attendance <75% + grade drop), fee-defaulter prediction, pacing alerts, leave anomalies — deterministic queries over data NEXLI already has, sitting as static panels. Split AI into "computed-now" vs "provider-later"; wire rule-based signals to live data today. The §11.1 differentiators that sell (Counselor Referral, Hardship Workflow, Exam-Stress Calendar, Night SOS) are workflow/trigger features, not AI, yet they're the moat. NEXLI built the ERP and stubbed the operating system.

The Outsider — fresh eyes

The parent can't message anyone — yet the spec's #1 promise is "message your class teacher first." Parent nav has "Communication" = a read-only announcements list; the two-way Messages engine exists only for staff. A parent asking "why was my daughter marked absent?" finds no reply button and returns to WhatsApp. "Communication" vs "Messages" is jargon (label "Notices/Circulars" and "Messages (Chat)"). Student support says "Message a teacher in Communications" — read-only, a dead end. Staff nav is a 35-item wall that doesn't feel role-shaped. "Class Assessments" vs "Holistic Progress Card" vs "Examinations" — three marks tabs. Fix parent/student messaging first.

The Executor — Monday morning

One blocker: any authenticated user can DevTools→Firestore→POCSO/payroll/Aadhaar/marks. Monday: (1) deploy/tighten firestore.rules today — per-tenant + lock confidential collections (free on Spark; the "P9" framing makes a present vuln sound future-tense). (2) Fix leaks rules can't catch — assessment_results peers' marks, event_registrations names — split into per-student docs (half a day). (3) Emulator-test vs teacher/parent/accountant before any human logs in. THEN role-test. Defer to Blaze: B1–B11, doc-numbering, pagination. A 1–3 day blocker, not a re-architecture.

Peer review highlights

Strongest: Executor ×4, Contrarian ×1 — most actionable; debunks the "future-tense" framing. Biggest blind spot: Expansionist ×3 / Outsider ×2 — auditing the cosmetic/AI/UX layer while the data layer is open ("optimizing layer 5 while layer 1 is breached"). All five missed: whether the data model can be scoped by rules (verified: yes, additive — no migration); the spec itself should be reconciled to Spark; nobody verified the rules deploy; the Blaze cost/viability ceiling for a DPDP multi-tenant OS.
